# 模型微调 — Fine-tuning

> 在预训练模型的基础上，用特定领域的数据进一步训练，提升模型在特定任务上的表现。

---

## 📋 概念速览

| 项目 | 内容 |
|------|------|
| 定义 | 在预训练模型基础上，用标注数据继续训练 |
| 与 RAG 的关系 | 互补：微调改进行为，RAG 注入知识 |
| 主流方法 | 全量微调、LoRA、QLoRA |
| 适用场景 | 风格迁移、指令遵循、特定格式输出 |

## 一、什么是微调

微调（Fine-tuning）是指在已预训练好的 LLM 基础上，使用特定领域的高质量标注数据继续训练，从而让模型适应特定任务或风格的过程。

### 微调 vs RAG vs Prompt Engineering

| 方法 | 优势 | 劣势 | 最佳场景 |
|------|------|------|---------|
| **Prompt Engineering** | 零成本、快速迭代 | 效果上限有限 | 简单任务、快速验证 |
| **RAG** | 知识实时更新、可追溯 | 需要检索基础设施 | 知识密集型任务 |
| **Fine-tuning** | 深度定制、效果稳定 | 成本高、需要训练数据 | 风格/行为定制 |

## 二、微调方法对比

### 2.1 全量微调 (Full Fine-tuning)

更新模型所有权重参数。

- **优点**：效果最佳，理论上限最高
- **缺点**：需要大量 GPU 显存（如 7B 模型需 ~56GB+），训练成本高
- **适用**：有充足算力资源，追求最佳效果

### 2.2 LoRA (Low-Rank Adaptation)

在模型权重旁附加小型可训练矩阵，仅更新附加参数。

- **优点**：显存需求降低 8-16 倍，训练速度快
- **缺点**：效果略低于全量微调
- **适用**：个人研究者、中小团队

### 2.3 QLoRA (Quantized LoRA)

在 LoRA 基础上对基础模型进行 4-bit 量化。

- **优点**：可在单张消费级 GPU（如 RTX 4090）上微调 65B 模型
- **缺点**：训练速度较慢
- **适用**：资源受限的微调场景

## 四、微调流程

```
1. 数据准备
   收集 → 清洗 → 格式化（如 Alpaca JSON 格式）
        ↓
2. 基线评估
   在目标任务上测试原始模型，建立基准
        ↓
3. 选择方法
   全量 / LoRA / QLoRA
        ↓
4. 训练
   设置超参数 → 开始训练 → 监控 loss
        ↓
5. 评估
   与基线对比 → 在测试集上验证
        ↓
6. 部署
   合并权重（LoRA）→ 导出 → 部署推理
```

## 五、最佳实践

1. **数据质量 > 数据数量**：500 条高质量数据好于 5000 条低质量数据
2. **保留测试集**：至少留 20% 数据用于评估
3. **避免灾难性遗忘**：混合通用数据与领域数据
4. **先试小模型**：在 1B-3B 模型上验证数据质量，再放大
5. **评估先行**：微调前先建立评估指标和基线
