# ✅ 验真系统说明

> **这是本项目的核心特色模块。** 在 AI 技术日新月异的时代，信息的真实性和时效性是最大的挑战。本验真系统旨在为每一条关键信息提供可追溯的验证。

---

## 一、为什么需要验真

AI 领域的信息面临三大问题：

```
1. 过时快 (Fast Expiry)
   模型一周一更新，API 一月一改版，文档跟不上变化
   
2. 真假难辨 (Hard to Verify)
   大量"二手信息"以讹传讹，AI 生成的虚假信息泛滥
   
3. 来源不明 (Unknown Source)
   很多资料不标注日期、不注明来源，无法判断可信度
```

**本项目的对策：为每条关键信息建立验真档案。**

## 二、验真等级定义

| 等级 | 标记 | 含义 | 示例 |
|------|------|------|------|
| **A 级** | ✅ 已完成验真 | 通过实际操作或官方文档验证 | API 申请流程、当前定价 |
| **B 级** | 🔍 交叉验证 | 多方来源交叉验证，未实际操作 | 框架对比、最佳实践 |
| **C 级** | 📋 理论整理 | 基于公开资料整理，未深度验证 | 概念介绍、原理说明 |
| **D 级** | ⏳ 待验真 | 已编写但未经过任何验证 | 新创建的内容 |
| **E 级** | ⚠️ 已过期 | 确认已过时，需要更新 | 已废弃的 API、旧版本功能 |

## 三、验真流程

### 标准验真流程

```
1. 确定验真对象
   └→ 文档 / 提示词 / Skills / API 信息
   
2. 设计验真方案
   └→ 明确要验证的点、验证方法、判断标准
   
3. 执行验真
   └→ 实际操作 / 查阅官方文档 / 交叉比对
   
4. 输出验真报告
   └→ 填写验真模板（见各子目录 template.md）
   
5. 更新文档状态
   └→ 将文档的 verification 字段改为 true
   └→ 添加验真链接指向报告
```

### 验真周期建议

| 内容类型 | 建议验真周期 | 说明 |
|---------|------------|------|
| API 定价 | 每月 | 价格变动频繁 |
| API 申请流程 | 每季度 | 流程相对稳定 |
| 模型能力基准 | 每月 | 新模型不断发布 |
| 概念原理 | 每半年 | 基础理论相对稳定 |
| 提示词效果 | 每次使用前 | 效果因模型版本而异 |
| Skills 可用性 | 每次使用前 | 依赖环境变化 |

## 四、验真报告结构

每份验真报告包含以下核心字段：

| 字段 | 说明 | 必填 |
|------|------|------|
| 验真对象 | 验证的具体内容 | ✅ |
| 验真日期 | 执行验真的日期 | ✅ |
| 验真方法 | 如何验证的（实际操作/文档核实/交叉验证） | ✅ |
| 验真结果 | 通过 / 部分通过 / 不通过 | ✅ |
| 效果评级 | A/B/C/D（见上表） | ✅ |
| 参考来源 | 验证依据的链接或截图 | ✅ |
| 验真人 | 执行验真的人员或系统 | ✅ |
| 备注 | 其他说明 | - |

## 五、验真标签

所有通过验真的文档将在头部添加标准标签块：

```markdown
---
verification: true
verified_date: 2026-05-07
verified_by: username
verified_method: 实际操作验证
verification_report: ../verification/prompts/xxx-report.md
rating: A
---
```

## 七、验真流程示例

以验证 OpenAI API 申请流程为例：

```markdown
验真对象：OpenAI API 注册流程
验真日期：2026-05-07
验真方法：实际操作注册流程
验真步骤：
  1. 访问 https://platform.openai.com/signup
  2. 使用 Google 账号注册
  3. 验证邮箱 ← 收到验证邮件（耗时约 30 秒）
  4. 添加手机验证
  5. 登录控制台
  6. 创建 API Key
验真结果：✅ 流程可用
效果评级：A
参考来源：实际操作录屏 /screenshots/openai-signup-20260507.png
备注：中国地区需要境外网络环境
```
