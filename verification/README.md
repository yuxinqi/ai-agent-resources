# 验真系统说明

> **这是本项目的核心特色模块。** 在 AI 技术日新月异的时代，信息的真实性和时效性是最大的挑战。本验真系统旨在为每一条关键信息提供可追溯的验证。

---

## 一、为什么需要验真

AI 领域的信息面临三大问题：

1. **过时快 (Fast Expiry)** — 模型一周一更新，API 一月一改版，文档跟不上变化
2. **真假难辨 (Hard to Verify)** — 大量"二手信息"以讹传讹，AI 生成的虚假信息泛滥
3. **来源不明 (Unknown Source)** — 很多资料不标注日期、不注明来源，无法判断可信度

**本项目的对策：为每条关键信息建立验真档案。**

---

## 二、证据等级 (Evidence Level)

证据等级表示信息的验证深度，从 L0（最高）到 L4（最低）：

| 等级 | 标记 | 含义 | 验证方法 | 示例 |
|------|------|------|---------|------|
| **L0** | 🟢 L0 | 官方文档直接验证 | 直接引用官方文档/API 响应 | API 端点、官方定价 |
| **L1** | 🔵 L1 | 实际操作验证 | 亲自执行操作并记录结果 | 注册流程、SDK 代码运行 |
| **L2** | 🟡 L2 | 交叉验证 | 多方来源交叉比对 | 框架对比、最佳实践 |
| **L3** | 🟠 L3 | 理论推导 | 基于已知信息逻辑推理 | 架构设计、趋势预测 |
| **L4** | 🔴 L4 | 待验证 | 已编写但未经任何验证 | 新创建的内容 |

### 证据等级的使用

- 所有文档应在 frontmatter 中标注 `evidence_level`
- 验真报告中必须标注证据等级
- 证据等级越高，信息可信度越高
- L4 级内容应优先安排验证

---

## 三、评级体系 (Rating)

评级表示信息的整体质量，从 A（最高）到 D（最低）：

| 评级 | 标记 | 含义 | 通过率 | 说明 |
|------|------|------|--------|------|
| **A** | ✅ A | 完全验证通过 | ≥90% | 可直接用于生产环境 |
| **B** | 🔍 B | 大部分验证通过 | ≥70% | 核心信息准确，细节可能需更新 |
| **C** | 📋 C | 部分验证通过 | ≥50% | 基础信息可用，需谨慎参考 |
| **D** | ⏳ D | 待验真 | <50% | 未经充分验证，仅供参考 |

### 评级与证据等级的关系

| 证据等级 | 可达最高评级 | 说明 |
|---------|------------|------|
| L0 | A | 官方验证，最高可信度 |
| L1 | A | 实际操作验证，高可信度 |
| L2 | B | 交叉验证，良好可信度 |
| L3 | C | 理论推导，一般可信度 |
| L4 | D | 待验证，低可信度 |

## 四、指标包 (Metrics Packs)

为不同类型的验证提供标准化的指标集：

### 4.1 提示词指标包

| 指标 | 说明 | 计算方法 |
|------|------|---------|
| 格式遵循率 | 输出是否符合指定格式 | 格式匹配 / 总测试次数 |
| 内容准确率 | 输出内容是否正确 | 正确输出 / 总输出 |
| 一致性 | 多次运行结果是否一致 | 一致次数 / 总运行次数 |
| 安全性 | 是否产生有害内容 | 安全输出 / 总输出 |

### 4.2 技能指标包

| 指标 | 说明 | 计算方法 |
|------|------|---------|
| 功能正确性 | 是否按预期工作 | 正确结果 / 总测试 |
| 触发准确性 | 是否在正确场景触发 | 正确触发 / 总触发 |
| 边界处理率 | 异常输入的处理能力 | 正确处理 / 异常输入数 |
| 兼容性 | 跨模型/环境表现 | 兼容模型数 / 测试模型数 |

### 4.3 平台指标包

| 指标 | 说明 | 计算方法 |
|------|------|---------|
| 信息准确率 | 与官方信息一致程度 | 一致项 / 总验证项 |
| 时效性 | 信息是否为最新 | 最新项 / 总项 |
| API 可用率 | API 是否可正常调用 | 成功调用 / 总调用 |
| 定价准确率 | 定价信息是否准确 | 准确项 / 总定价项 |

### 4.4 工作流指标包

| 指标 | 说明 | 计算方法 |
|------|------|---------|
| 任务完成率 | 能否完成目标任务 | 完成任务 / 总任务 |
| 步骤效率 | 实际步骤 / 最优步骤 | 最优步骤 / 实际步骤 |
| 输出质量 | 最终输出的质量 | 评分 / 满分 |
| 成本效率 | Token 消耗合理性 | 实际消耗 / 预期消耗 |

## 五、验真流程

### 标准验真流程

1. **确定验真对象** — 选择需要验证的文档/内容
2. **确定目标证据等级** — 根据内容类型设定目标等级
3. **设计验真方案** — 明确验证方法、测试用例、判断标准
4. **执行验真** — 按方案执行验证，记录所有过程和结果
5. **输出验真报告** — 使用统一模板记录结果
6. **更新文档状态** — 更新 frontmatter 中的验真字段

### 验真周期建议

| 内容类型 | 建议验真周期 | 目标证据等级 | 说明 |
|---------|------------|------------|------|
| API 定价 | 每月 | L0-L1 | 价格变动频繁 |
| API 申请流程 | 每季度 | L1 | 流程相对稳定 |
| 模型能力基准 | 每月 | L2 | 新模型不断发布 |
| 概念原理 | 每半年 | L2-L3 | 基础理论相对稳定 |
| 提示词效果 | 每次使用前 | L1-L2 | 效果因模型版本而异 |
| 工作流 | 每季度 | L2 | 依赖组件变化 |

## 六、验真报告结构

每份验真报告应包含以下核心字段：

| 字段 | 说明 | 必填 |
|------|------|------|
| 基本信息 | 验真对象、日期、方法、证据等级 | ✅ |
| 验真结论 | 结果、评级、总体评价、改进建议 | ✅ |
| 测试任务 | 验证的测试任务列表 | ✅ |
| 测试输入 | 具体的测试输入数据 | ✅ |
| 实际输出 | 测试的实际输出 | ✅ |
| 指标结果 | 量化指标和数值 | ✅ |
| 成功情况 | 验证通过的部分 | ✅ |
| 失败情况 | 验证未通过的部分 | ✅ |
| 成本与延迟 | Token 消耗和响应延迟 | ✅ |
| 风险与限制 | 已知的风险和局限性 | ✅ |
| 复现步骤 | 如何复现验真过程 | ✅ |
| 后续复验计划 | 下次复验的时间和重点 | ✅ |
